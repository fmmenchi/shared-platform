# Core Concepts

Why this plugin exists, and how it assembles one site from many `docs/` folders.

---

## 💡 The Philosophy

### 1. Docs live in the package, not in a separate site

Each package's documentation lives in a `docs/` folder next to its code. A change to the code and
the change to its docs land in the **same package, the same PR** — there is no far-away docs tree to
forget. Adding docs to a new library is zero-config: ship a `docs/` folder and the plugin discovers
it from the Nx graph on the next build.

### 2. Author raw Markdown, let the engine publish

Authoring and publishing are decoupled. You write CommonMark; the executors handle discovery,
categorisation, and assembly. The site is a thin non-published app under `apps/` whose only job is
to render the assembled tree — so it renders with plain Docusaurus: native sidebar behaviour, no
custom rendering layer.

### 3. One site, libraries and plugins side by side

Workspace-wide docs (architecture, ADRs, styling) are authored directly in the site's own
co-located `docs/`. Per-package docs are assembled in beside them under **Libraries** and
**Plugins**. A reader never has to know whether something is a library or a plugin to find it —
it is one navigation tree.

### 4. Assemble, never duplicate in git

The assembled per-package folders are a **copy**, rebuilt on every sync. Committing them would
duplicate the source and invite drift, so `sync-docs` writes a `.gitignore` that keeps them out of
version control. Only the source `docs/` folders and the `_category_.json` sidebar markers are
tracked.

---

## 🏗 How It Works Inside

The site is built by two executors that run **before** Docusaurus, wired as dependencies of the
site's `build` and `start` targets. The Nx project graph is the discovery mechanism.

### The assembly pipeline

1. **`config-generator` — the mapper.** Runs in the site project's context and walks
   `context.projectsConfigurations.projects`. For every project it skips the docs site itself and
   any `application`, then keeps the ones that ship a `docs/` folder containing at least one
   `.md`/`.mdx` file or a `_category_.json`. Each kept project becomes a manifest entry — its Nx
   `name`, `root`, an unscoped destination `folder` (`@fmmenchi/notify` → `notify`), and a `type`
   of `plugin` (tag `scope:plugins`) or `library`. The manifest, `nx-doc-projects.json`, is written
   into the site project's root.

2. **`sync-docs` — the worker.** Reads `nx-doc-projects.json` and copies each project's `docs/` into
   `<targetPath>/{libraries,plugins}/<folder>`, replacing whatever was there. It also writes
   `<targetPath>/.gitignore` (`libraries/*/`, `plugins/*/`) so the assembled copies stay out of git.
   With `watch: true` it stays running as a continuous executor, watching every source `docs/`
   folder and re-syncing the changed project — this is what the dev server depends on for hot
   reload.

3. **Docusaurus — the renderer.** Once the tree is assembled, `docusaurus build` / `docusaurus
start` run over a flat local `docs/` folder like any ordinary site. The sidebar is autogenerated
   from that folder structure and the `_category_.json` markers.

### The self-contained tree invariant

Because the assembled tree is a single flat folder, every in-site link must resolve **within it** —
cross-package links point into the assembled paths (e.g. `../../plugins/nx-notify/index.md`). The
generated site sets `onBrokenLinks: 'throw'`, so a dead link fails the build rather than shipping
silently. Destination folders use the **unscoped** package name to stay unique and collision-free.

### Where the pieces live

- Source of truth: each package's own `docs/` folder (tracked).
- Workspace docs + sidebar markers: the site's co-located `docs/` and its `docs/{libraries,plugins}/_category_.json` (tracked).
- Assembled per-package folders + `nx-doc-projects.json`: generated, git-ignored.

See the [reference](../reference/index.md) for the exact options, and the
[build & serve guide](../guides/build-and-serve.md) for how the targets chain together.
